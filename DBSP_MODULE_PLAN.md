# DBSP Module Improvement Plan
_Generated by dbsp-module-team — 4 parallel specialist agents_

---

## Executive Summary

The `spooky_streaming_processor` is a well-structured proof-of-concept for an incremental streaming query engine. The push-based DAG pattern, symmetric hash join, and decorrelation logic are all architecturally sound. However, the codebase has **8 critical correctness bugs** — not edge cases, but fundamental flaws that produce wrong output for any non-trivial query. The most severe: `JoinType::LeftSemi` silently executes as an inner join; weight propagation in join outputs ignores stored record weights; and the lowering pass silently discards any WHERE clause it doesn't recognise. Before any performance work is valid, these correctness gaps must be closed. Three agents independently confirmed the same top issues, raising their priority.

**Severity distribution**: 8 critical bugs · 12 Phase 1 items · 9 Phase 2 items · 7 Phase 3 items · 10 Phase 4 items.

**Top 3 priorities**: (1) silent `_ => scan` fallback in lowering.rs makes any unrecognised condition invisible; (2) `JoinType::LeftSemi` ignored by the engine — all subquery-decorrelated joins run as inner joins; (3) `diff: i32` overflows at batch size ~46k, corrupting all output weights silently.

## Critical Bugs (Fix Immediately — Correctness Risk)

| # | Bug | File:Line | Impact | Fix |
|---|-----|-----------|--------|-----|
| C1 | `_ => scan` silent swallow | `src/lowering.rs:143` | Any unrecognised WHERE clause becomes a full-table scan. Compound AND/OR, column-to-column comparisons, future operators — all silently dropped. User sees unfiltered results with no error. | Replace with `Err(LoweringError::UnsupportedCondition { description })` — confirmed by agents 1, 2, 4. |
| C2 | `JoinType::LeftSemi` ignored by engine | `src/engine/builder.rs:64` + `src/engine/join.rs:96-111` | `IN (subquery)` queries always execute as inner joins. Returns duplicate left rows when right side has multiple matches — violates set semantics entirely. | Pass `kind` through `JoinCore`; in Phase 1 probe emit at most one output per left record for `LeftSemi`. Confirmed by agents 1, 4. |
| C3 | Join weight propagation ignores stored weight | `src/engine/join.rs:108` | `diff: event.diff` copies only the arriving event's weight. Correct DBSP formula: `output_weight = arriving_weight × stored_weight`. Any pipeline with non-unit weights (batching, chained joins) produces wrong output. | Store `(i64, Record)` pairs in `left_map`/`right_map`; output `diff: event.diff * stored_weight`. Confirmed by agents 1, 3. |
| C4 | `diff: i32` overflow | `src/engine/mod.rs:21` | Weight product of two 46k-record batch inserts: `46_341 × 46_341 = 2_147_488_281 > i32::MAX`. Overflows silently in release mode, corrupting all output weights. | Change to `pub diff: i64`. Zero-cost on x86_64. Confirmed by agents 1, 3, 4. |
| C5 | `SurrealOp::Inside => Predicate::Eq` in scalar arm | `src/lowering.rs:134` | `col INSIDE 5` is silently lowered to `Predicate::Eq(col, 5)` — membership test against a scalar, which is semantically invalid. | Return `Err(LoweringError::InsideRequiresSubquery { col })` when `Inside` appears with a `Literal` on the right. Confirmed by agents 2, 4. |
| C6 | `_ => false` in `Predicate::eval` | `src/plan.rs:34` | Any future `Predicate` variant (`Ne`, `And`, `Ge`) compiles and type-checks but evaluates to `false` for all records — silently producing empty output instead of a compile error. `Gt`/`Lt` with `Value::Str` or `Value::Bool` also silently return false. | Delete the `_ => false` arm entirely; Rust exhaustiveness will catch any unhandled variant as a compile error. Confirmed by agents 1, 2, 4. |
| C7 | `build_graph` panics on non-Scan join inputs | `src/engine/builder.rs:67-70` | `extract_scan_table(&left).expect(...)` panics on any `LogicalPlan::Join` whose child is a nested join, Union, Aggregate, or any future non-Scan root. Crashes the process on user input. | Change signature to `Result<Box<dyn Operator>, BuildError>`; use `.ok_or_else(|| BuildError::NoScanInSubtree { side, plan_debug })?`. Confirmed by agents 1, 3, 4. |
| C8 | `JoinCore` maps never prune retractions | `src/engine/join.rs:119-134` | Every `diff: -1` event inserts the retracted `Record` as a new entry in `left_map`/`right_map`. The Vec grows without bound. Over time, probe cost becomes O(all historical events) per key. | Store `HashMap<Value, Vec<(Record, i64)>>`. In Phase 2, update weight in-place and prune entries where weight == 0: `bucket.retain(|(_, w)| *w != 0)`. Confirmed by agent 1. |

### 2.2 Add optimization passes

**New file**: `src/optimizer.rs`

Priority order (most value, least complexity first):

1. **Predicate pushdown**: `Filter(Join(A, B), p)` → `Join(Filter(A, p), B)` when `p` only references columns from `A`. Reduces rows entering the expensive Phase-1 probe.
2. **Projection pushdown**: push `Project` nodes toward the leaves to reduce column count early, reducing HashMap allocation size in join probes.
3. **Trivial project elimination**: `Project(plan, all_columns)` → `plan` (no-op removal).

```rust
pub fn optimize(plan: LogicalPlan) -> LogicalPlan {
    let plan = push_filters_below_joins(plan);
    let plan = prune_unused_columns(plan);
    eliminate_trivial_projects(plan)
}
```

### 2.3 Introduce `Weight` newtype

pub struct Weight(pub i64);

impl Weight {

    pub const DELETE: Self = Weight(-1);
    pub fn scale(self, factor: i64) -> Self { Weight(self.0 * factor) }

## Phase 3 — Performance

All performance items require benchmarks to be established first. See Benchmark Strategy section.

### 3.1 `Arc<Record>` to eliminate O(NM) clone-per-match

**File**: `src/engine/join.rs:19-21`, `src/engine/join.rs:103-110`

Current cost: **~2NM heap allocations per `on_event`** for N right-side matches and M columns. Dominant hotspot.

Change `JoinCore` maps to store `Arc<Record>`:

```rust
pub left_map:  HashMap<Value, Vec<(i64, Arc<Record>)>>,
pub right_map: HashMap<Value, Vec<(i64, Arc<Record>)>>,
```

Phase 2 wraps in Arc once at insert: `Arc::new(event.record)` — one allocation.
Phase 1 probe clones the Arc (atomic refcount bump, ~3 ns) instead of deep-copying all M columns.

Expected gain: 3–10× reduction in allocations on the join hot path at N=10 right matches.

Benchmark first: `cargo bench --bench join_probe` at N=1,10,100 right matches.

### 3.3 Batched ZSet processing mode (`BatchingOp`)

**New file**: `src/engine/batching.rs`

DBSP math derivation: with consolidation rate `c` (fraction of batch that cancels):
- Event-at-a-time work: proportional to `B` events
- Batched work: proportional to `B × (1 - c)`
- Gain: `1/(1-c)` — at c=50% = 2×, at c=90% = 10×

```rust
pub struct BatchingOp {
    buffer: HashMap<Record, i64>, // ZSet<Record>
    flush_threshold: usize,
    next: Box<dyn Operator>,
    source: String,
}

impl Operator for BatchingOp {
    fn on_event(&mut self, event: Event) {
        *self.buffer.entry(event.record).or_insert(0) += event.diff;
        if self.buffer.len() >= self.flush_threshold { self.flush(); }
    }
}
impl BatchingOp {
    fn flush(&mut self) {
        for (record, weight) in self.buffer.drain() {
            if weight != 0 {
                self.next.on_event(Event { source: self.source.clone(), record, diff: weight });
            }
        }
    }
}
```

Wire as a wrapper before `DispatcherOp`. Requires Phase 1 items 1.4 (i64 diff) and C3 (correct weight arithmetic) first.

Benchmark first: `cargo bench --bench batch_consolidation` to confirm break-even at c ≥ 10%.

## Brainstorm Items (Design Decisions Needed)

See `DBSP_BRAINSTORM.md` for full analysis. The following require a user decision before implementation:

1. **ZSet batch type strategy** — Keep `Event`-at-a-time only, or add `ZSet<Record>` batch layer alongside it? (See BRAINSTORM §1)
2. **Weight type** — `i64` bare integer vs `Weight(i64)` newtype? The newtype prevents `weight * weight` (dimensionally invalid) but costs migration effort. (See BRAINSTORM §2)
3. **Optimization layer** — Rule-based heuristics vs Volcano cost model? The former is sufficient for the current SurrealQL query set. (See BRAINSTORM §3)
4. **Real SurrealDB AST** — Replace mock types now or after Phase 1 is complete? (See BRAINSTORM §4)
