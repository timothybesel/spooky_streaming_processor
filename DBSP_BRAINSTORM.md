_Generated by dbsp-module-team вЂ” design alternatives requiring user decisions_

## 1. ZSet Type Strategy

### Options

**A. Keep `Event`-at-a-time only (current)**
- `Event { record, diff: i64 }` is called once per record
- Simple; works for all existing demos
- Cannot consolidate retractions before expensive operators
- A `+1` and `-1` for the same record в†’ 2 full join probes, not 0

**B. Add `ZSet<Record>` batch layer alongside `Event` (recommended)**
- `Event` stays for compatibility
- Add `on_batch(&mut self, source: Arc<str>, batch: ZSet<Record>)` to the `Operator` trait with a default implementation that splits to `on_event`
- Operators that benefit (JoinSideOp, FilterOp) implement `on_batch` for consolidation
- `BatchingOp` wrapper accumulates events into a `ZSet<Record>`, consolidates, then flushes

```rust
pub type ZSet<K> = HashMap<K, i64>;  // K в†’ net weight

pub trait Operator {
    fn on_event(&mut self, event: Event);

    // Default: split batch into individual events (backwards compatible)
    fn on_batch(&mut self, source: Arc<str>, batch: ZSet<Record>) {
        for (record, weight) in batch {
            if weight != 0 {
                self.on_event(Event { source: source.clone(), record, diff: weight });
            }
        }
    }
}
```

**C. Replace `Event` entirely with `ZSet<Record>` (aggressive)**
- Pipeline operates on batches exclusively
- Eliminates per-event overhead entirely
- High migration cost вЂ” all existing operators must be rewritten
- Latency increases (must wait for batch to fill before processing)

**Tradeoffs**

| Option | Latency | Throughput | Migration Cost | Correctness |
|--------|---------|-----------|----------------|-------------|
| A (current) | Lowest | O(NM) allocs/event | 0 | Missing weight product |
| B (hybrid) | Low | Up to 100Г— at c=90% | Medium | Full after Phase 1 |
| C (batch only) | Medium | Maximum | High | Requires full rewrite |

**Recommendation**: Option B. Add `ZSet<Record>` as a parallel path via `on_batch`. Option A continues to work for all existing callers. Option C can be pursued later if the benchmarks from Phase 3.3 show batch-mode dominance is worth the latency trade.

## 2. Weight Type

### Options

**A. `diff: i64` bare integer**
- Minimal change from current `i32`
- No compile-time prevention of dimensional errors
- `event.diff * other.diff` (wrong: weight squared) compiles silently

**B. `Weight(i64)` newtype (recommended)**
- Implements `Add`, `Neg` but NOT `Mul<Weight, Weight>`
- `event.diff * other.diff` в†’ compile error: "cannot multiply Weight by Weight"
- Join weight product expressed as `Weight(event.diff.0 * stored_weight.0)` вЂ” intentionally verbose; makes the bilinear operation explicit
- `.scale(n: i64) -> Weight` for scalar multiplication

**C. Generic `W: WeightType` (over-engineered for now)**
- Allows `Weight = i32` for unit-weight pipelines, `Weight = i64` for general
- Adds type parameter to all operators and `Event`
- Maximum flexibility but high cognitive overhead; adds complexity before any need is demonstrated

**Recommendation**: Option B. The `Weight(i64)` newtype has zero runtime cost, prevents a class of correctness bugs at compile time, and documents intent clearly. Generic weights (C) can always be added later if multiple weight domains are needed.

```rust
// src/engine/weight.rs
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, PartialOrd, Ord)]
pub struct Weight(pub i64);

impl Weight {
    pub const INSERT: Self = Weight(1);
    pub const DELETE: Self = Weight(-1);
    /// Scale by a plain integer вЂ” e.g., for batched insertions.
    pub fn scale(self, factor: i64) -> Self { Weight(self.0 * factor) }
    pub fn is_zero(self) -> bool { self.0 == 0 }
    /// Bilinear join weight product вЂ” explicit, not via Mul<Weight, Weight>.
    pub fn join_product(left: Weight, right: Weight) -> Weight {
        Weight(left.0 * right.0)
    }
}

impl std::ops::Add for Weight  { type Output = Self; fn add(self, r: Self) -> Self { Weight(self.0 + r.0) } }
impl std::ops::Neg for Weight  { type Output = Self; fn neg(self) -> Self { Weight(-self.0) } }
// Intentionally no: impl Mul<Weight, Weight> for Weight
```

## 3. Optimization Layer

### Options

**A. No optimizer (current state)**
- `convert_to_plan` в†’ `build_graph` with no intervening pass
- Plans are never rewritten; filter order matches source order
- For complex queries (3+ tables, nested subqueries) the engine may probe unnecessarily large tables

**B. Rule-based heuristics (recommended for current scope)**
- A set of algebraic equivalence rules applied top-down, fixed-point iteration
- Rules: predicate pushdown, projection pushdown, filter merging, trivial project elimination
- No statistics required; all rules are correctness-preserving by construction
- Sufficient for all SurrealQL queries in `query.rs`

```rust
pub fn optimize(plan: LogicalPlan) -> LogicalPlan {
    let rules: &[fn(LogicalPlan) -> LogicalPlan] = &[
        push_filters_below_joins,
        prune_unused_columns,
        eliminate_trivial_projects,
    ];
    let mut current = plan;
    loop {
        let next = rules.iter().fold(current.clone(), |p, rule| rule(p));
        if next == current { break; }
        current = next;
    }
    current
}
```

Requires `LogicalPlan: Clone + PartialEq` (Phase 1 item 1.2).

**C. Volcano/Cascades cost-based optimizer**
- Each operator has a cost formula; optimizer searches the space of equivalent plans for minimum cost
- Requires cardinality statistics (row counts, column distributions) from a catalog
- Significant complexity; typical implementation is 2,000вЂ“5,000 lines of Rust
- Justified only when join reordering matters (3+ table joins with skewed cardinalities)

**D. Hybrid: rule-based first, cost-based for join reordering**
- Apply rules B unconditionally; add a greedy cardinality-aware join reorder on top
- A reasonable upgrade path from B to C without building a full Volcano planner

**Recommendation**: Option B now, with Option D as the next step when multi-table joins (3+ tables) are common. The incremental streaming model gives free predicate pushdown semantics (filters close to the source table reduce join probe work most); rule-based is sufficient for this.

**Value of each rule for SurrealQL queries**:

| Rule | Value | Complexity |
|------|-------|------------|
| Predicate pushdown | High вЂ” reduces join probe input size | Low (~50 LOC) |
| Projection pushdown | Medium вЂ” reduces record width in probes | Low (~50 LOC) |
| Filter merging | Low вЂ” collapses adjacent Filters | Trivial (~20 LOC) |
| Join reordering (greedy) | High for 3+ tables | Medium (~200 LOC) |
| Join reordering (DP) | High for 5+ tables | High (~500 LOC) |

## 4. Real SurrealDB AST vs Mock

### Options

**A. Keep mock types (current)**
- `SurrealSelectStmt`, `SurrealCond`, `SurrealExpr`, `SurrealOp` in `lowering.rs`
- Simple, no dependency on `surrealdb_core` internals
- Diverges further from reality with every new query type
- Does not support graph traversal, parameters, FETCH, SPLIT, or any advanced SurrealQL feature

**B. Wrap real types with a translation layer**
- Write `fn lower_select(stmt: surrealdb_core::sql::SelectStatement) -> Result<LogicalPlan, LoweringError>`
- Keep mock types as test helpers internally; mark them `#[cfg(test)]`
- `surrealdb-core` is already in `Cargo.toml`

**C. Replace entirely with real types**
- Delete all mock types; only accept real `surrealdb_core::sql` types at the API boundary
- Maximum accuracy; no translation layer drift
- Requires implementing the full operator coverage table from the lowering skill

**Migration complexity**

| Gap | Mock | Real | Work |
|-----|------|------|------|
| `op: MoreThanEqual` | Missing | Present | Add `SurrealOp::Ge` or use real type |
| `Literal(Integer(18))` | `Value::Int(i64)` | `Number::Int(i64)` | One mapping layer |
| `Idiom([Field("age")])` | `Column(String)` | Nested `Idiom` + `Part::Field` | `idiom_to_column()` helper |
| Nested subqueries in SELECT | Not representable | `Subquery::Select(SelectStatement)` | Recursive lowering |
| `$param` references | Not representable | `Value::Param(Param)` | Binding map |
| Graph traversal `->` | Not representable | `Part::Graph(Graph)` | New `LogicalPlan` node |

**Recommendation**: Option B. `surrealdb-core 3.0.0` is already declared. Add `real_lowering.rs` that accepts real types and calls the existing (fixed) plan-construction logic. Keep mock types for unit tests вЂ” they are simpler to construct than real AST nodes. The real lowering layer will expose gaps in operator coverage that must be addressed incrementally.

## 5. Parallel Execution

### Options

**A. Stay `Rc<RefCell>` вЂ” single-threaded (current, recommended for now)**
- No synchronisation overhead
- All operators are `!Send`; the constraint is documented but not enforced at compile time
- Fine for a single-core streaming processor or an embedded query engine

**B. `Arc<Mutex<JoinCore>>` вЂ” shared mutable state**
- Allows multiple threads to push events; each `on_event` acquires a Mutex
- Mutex contention on the join hash table is the bottleneck вЂ” effectively serialises join operations
- No real parallelism gained for join-heavy pipelines

**C. Channel-based pipeline parallelism**
- Each operator runs in its own thread; events flow over MPSC channels between stages
- Filter and Project stages can run truly in parallel with the join's probe phase
- Increases latency (channel overhead) but increases throughput for pipeline-parallelisable plans
- Operators become `Send`; eliminates `Rc`; state is owned per-thread

**D. Rayon `par_iter` over batches**
- Multiple batches processed in parallel across the Rayon thread pool
- Requires `on_batch` (ZSet layer from В§1 Option B) to be `Send`
- Simple to add once the ZSet batch layer exists
- Does NOT parallelise within a single batch

**DBSP parallel execution note**: DBSP's theoretical model supports *data parallelism* by partitioning the input ZSet across workers and merging output ZSets. This is more principled than pipeline parallelism and directly applicable if ZSet batches are introduced (В§1 Option B).

**When would parallelism actually help?**
- Only when individual operator processing cost > inter-thread communication cost (~1вЂ“10 Вµs)
- For the current workload (in-memory hash probes at ns-scale), threading overhead likely exceeds gain
- Measure with `bench_join_probe` at 10k events before adding any threading

**Recommendation**: Stay with Option A until benchmarks show single-core saturation. Then evaluate Option D (Rayon over ZSet batches) as the lowest-risk parallel option, since it requires only a `Send` bound on `ZSet<Record>` rather than restructuring the entire operator graph.

## 6. Graph Traversal Support

SurrealQL's defining feature: `SELECT ->knows->person FROM user:alice`.

### Options

**A. Return `LoweringError::UnsupportedGraphTraversal` (recommended for now)**
- No implementation required
- Clean error message; user knows to reformulate the query
- Unblocks Phase 1 work without the complexity of graph operators

**B. Rewrite to a JoinChain at the lowering layer**

```
user:alice ->knows->person
в†’ Scan(knows)
  + Filter(in = "user:alice", Scan(knows))
  + Join(person, on: knows.out = person.id)
```

Each `->edge->` hop becomes a two-step join. Works for simple single-hop traversals. Multi-hop becomes a chain of `Join` nodes вЂ” supported once Phase 2 multi-level join is implemented.

**C. Dedicated `GraphTraversal` node in `LogicalPlan`**

```rust
LogicalPlan::GraphTraversal {
    start: Box<LogicalPlan>,    // base table / record filter
    hops: Vec<GraphHop>,         // each hop: Dir Г— edge_table Г— dest_table
    alias: Option<String>,
}
```

Most flexible; enables path queries, variable-depth traversals (`->knows*..5->`), and graph-specific optimizations. High implementation cost; requires new physical operator and integration state.

**D. Delegate to SurrealDB engine**
- Pass graph queries through to the underlying SurrealDB instance
- No implementation needed
- Loses incremental streaming semantics for graph results

**Recommendation**: Option A (return error) until single-hop graph queries are needed. Then Option B (rewrite to JoinChain) вЂ” it reuses existing `Join` infrastructure and handles the majority of real-world graph traversal patterns. Option C is the correct long-term design but is large scope.

**Pre-requisites for Option B**:
- Phase 1.2 (multi-level join support in `build_graph` вЂ” removes the single-Scan constraint)
- Phase 2.1 (PhysicalPlan layer вЂ” `GraphHop` physical node)
- Record IDs in `Value` (e.g., `Value::Thing(table, id)`)

## 7. ZSet Key Design for a KV-backed Engine

### The Question

When a key-value store (e.g. SurrealDB, RocksDB) holds records like:

```
user:sdf8sd9f8 => { name: "timothy", age: 28 }
```

вЂ¦what should the **ZSet element** (the K in `Z[K] : K в†’ в„¤`) be when the KV store
emits a change event?

Three options were proposed and evaluated against DBSP algebra, join correctness,
filter pushdown, and cache performance.

---

### Option A вЂ” Record ID as ZSet element

```
ZSet entry: (Value::Str("user:sdf8sd9f8"), +1)
```

The event carries only the record identifier. Field values are not in the ZSet element.

**DBSP correctness: вќЊ Fatally wrong.**

The ZSet element must uniquely identify a distinct relational tuple including its field
values. An ID alone does not. Two records `{id:"X", age:28}` and `{id:"X", age:29}`
are **different tuples in the relational bag** вЂ” they must map to different ZSet elements.
With Option A, both map to the same element `"X"`.

Consequences:
- **Update is invisible.** When age changes 28 в†’ 29, the CDC produces two identical
  ID-only events: `("X", -1)` and `("X", +1)`. `BatchingOp`'s equality check finds a
  match, increments net weight by -1 then +1 в†’ zero в†’ **pruned**. The update is
  silently swallowed before it reaches any downstream operator.
- **Filter breaks.** `FilterOp` calls `record.get("age")` on the incoming event. With
  Option A the record is `{"id": "user:sdf8sd9f8"}` вЂ” no `age` field.
  `Predicate::Gt("age", ...)` hits `_ => false` and every record is silently discarded.
- **Join output is empty.** The merged record after a join probe contains only the ID
  from the left side plus whatever the right side stored (also ID-only). No user-facing
  columns appear in the output.

**Verdict: Definitively wrong. Do not use.**

---

### Option B вЂ” Full record as ZSet element (current approach)

```
ZSet entry: ({id:"user:sdf8sd9f8", name:"timothy", age:28}, +1)
```

The full denormalized record is the ZSet element. For `SELECT name, age FROM users`,
`ProjectOp` trims to:

```
ZSet entry: ({name:"timothy", age:28}, +1)
```

**DBSP correctness: вњ… Correct.**

`Z[Record] : Record в†’ в„¤` satisfies every ZSet axiom:
- Element identity = full tuple в†’ distinct field values = distinct elements.
- Weights are integers in в„¤; pointwise addition is well-defined.
- Retraction: CDC emits `(old_record, -1)` then `(new_record, +1)`. Because
  `old_record в‰  new_record` (they differ in the changed field), they are separate ZSet
  entries and both propagate downstream independently.

**Join probe:** `event.record.get(join_key_col)` extracts the join column value.
`merged.extend(other.as_ref().clone())` produces the full combined tuple with all
columns from both sides. The current `JoinMap = HashMap<Value, Vec<(Arc<Record>, i64)>>`
is already this design:

```
outer key: join column value  в†’ fast O(1) index (not the ZSet key)
inner vec: (Arc<Record>, i64) в†’ the ZSet entries (full record = ZSet key)
```

The `Arc<Record>` stored in the bucket is the **same heap allocation** as the incoming
event record (after `Arc::new(event.record)` on join.rs:201). No duplicate copy.
Refcount clones during Phase 1 probe are pointer increments, not record copies.

**Filter pushdown:** `record.get("age")` hits the in-memory `HashMap` in ~5вЂ“15 ns.
All column values are co-located in the same `HashMap` backing table (~3 cache lines
for a 5-column record). Filter + join key extraction + projection all touch the same
hot cache lines вЂ” L1-friendly.

**Performance:**
- ~500 bytes/event for a 5-column record (48-byte HashMap + backing table + heap strings)
- `JoinMap` overhead: ~56 bytes of Arc/metadata per stored record (record itself shared)
- At 1M records: ~556 MB total (events transient; JoinMap retains only live records)
- Update path: `Vec::find()` comparison is O(M Г— N_cols) where M = records per bucket.
  For unique join keys (M=1, the common case) this is O(N_cols) в‰€ 5 comparisons вЂ” fast.

**Projection:** `ProjectOp` places `HashMap::retain` over N columns keeping K.
At 5 total / 2 retained: ~10 string comparisons + 3 heap drops per event (~80 ns).
For 1M events: ~80 ms total вЂ” acceptable.

**Invariant the design must maintain:**
> `ProjectOp` must never strip the join-key column before that column reaches
> `JoinSideOp`. If the optimizer places `Project` before `Join` and projects away
> the join key, retraction events arrive at `JoinSideOp` with `record.get(join_key) == None`,
> breaking both probe and retraction. The optimizer must enforce that join-key columns
> survive until after the join materialises.

**Verdict: The only correct choice for this engine. Already implemented.**

---

### Option C вЂ” Per-column inverted ZSets

```
zset_name: { "timothy" => ["user:sdf8sd9f8"] }
zset_age:  { 28        => ["user:sdf8sd9f8"] }
```

Weight derived from `zset_age[28].len()` в†’ "how many records have this value".

**DBSP correctness: вќЊ Fatally wrong as a ZSet.**

`len()` is **not** a DBSP weight. DBSP weights are integers in в„¤ вЂ” they must support
additive inverses (retractions). `len()` is always `в‰Ґ 0`; you cannot represent `len = -1`.

Specific failures:
- **Retraction algebra broken.** After inserting users A and B both with `age=28`,
  `zset_age[28].len() = 2`. Deleting user A gives `len = 1`. From a DBSP perspective
  the delete of A should emit a `(record_A, -1)` event. With Option C, downstream
  sees only "cardinality of age=28 decreased by 1" вЂ” it cannot identify **which** record
  was retracted, so it cannot retract the corresponding join output.
- **Tuple identity lost.** The element `28` represents all records with `age=28`, not
  any specific tuple. Two distinct tuples `{id:X, age:28}` and `{id:Y, age:28}` collide
  into the same ZSet element.
- **Join incompatible.** `JoinSideOp` expects `Vec<(Arc<Record>, i64)>` вЂ” full records
  with weights. Reconstructing a full record from per-column ZSets requires N separate
  HashMap lookups, one per column. For 1M records Г— 5 columns with ZSet tables of
  ~72 MB each, this is a cache-cold random-access storm: ~500 ns per reconstructed
  record vs ~15 ns for Option B.
- **Filter incompatible.** `FilterOp::on_event` calls `record.get("age")` on the
  arriving event. Option C events do not carry field values in this form вЂ” it requires
  a completely different execution model (bulk set-intersection, not push-based row filter).

Option C is a legitimate **secondary data structure** (a secondary index or an inverted
index for accelerating specific predicates) but cannot serve as the primary ZSet
representation. It may be useful as an optimisation for range-scan predicates on
high-cardinality columns *after* the correct primary ZSet (Option B) is established.

**Verdict: Not a ZSet. Incompatible with every current operator. Do not use as primary representation.**

---

### Option D вЂ” Hybrid (current design, recommended)

```
JoinMap = HashMap<join_key_value, Vec<(Arc<Record>, i64)>>
```

- **Outer HashMap key** = join column value (like Option A) в†’ O(1) probe index
- **Inner Vec entries** = `(Arc<Record>, i64)` = full-record ZSet elements with weights (Option B)

This is exactly what `src/engine/join.rs` already implements. The outer key is a
**secondary index** for fast probe lookup, not the ZSet identity key. The ZSet identity
is the full record, preserved via full `HashMap<String, Value>` equality in `find()`.

This design is algebraically correct and cache-efficient:
- Probe: O(1) outer lookup + O(M) inner scan (M = records per bucket; usually 1)
- Retraction: `find()` uses full record equality в†’ exact match, exact weight decrement
- Memory: one heap allocation per live record, shared by `Arc` between the event and
  the JoinMap entry вЂ” no double copy

**Verdict: The correct design. Already implemented.**

---

### CDC Source Requirements

For all correct options (B and D), the KV store CDC layer **must provide the old
record on delete/update**, not just the record ID:

| CDC event type | Required payload | DBSP delta emitted |
|----------------|------------------|--------------------|
| INSERT | `new_record` | `(new_record, +1)` |
| DELETE | `old_record` (full fields) | `(old_record, -1)` |
| UPDATE | `old_record` + `new_record` | `(old_record, -1)` then `(new_record, +1)` |

Without the old record at delete/update time, the `-1` event cannot be emitted with the
correct ZSet element, and the downstream operators cannot retract their previous outputs.
This is a **source adapter requirement**, not an engine requirement. SurrealDB's
LIVE SELECT, RocksDB's column family column families, and Postgres logical replication
(with `REPLICA IDENTITY FULL`) all provide this.

## 8. WHERE / ORDER BY / LIMIT вЂ” Separate Nodes vs Pushed Into Scan

_Generated by dbsp-module-team brainstorm вЂ” 4 specialist agents (DBSP math, AST lowering, performance, clean code)_

### The Question

Currently the parser only handles `SELECT fields FROM table` (with subqueries).
The user wants to add WHERE, ORDER BY, and LIMIT. Three approaches were evaluated:

- **Approach A**: Separate nodes in LogicalPlan (current backup design)
- **Approach B**: Rich Scan node that absorbs filter/sort/limit
- **Approach C**: Hybrid вЂ” separate in LogicalPlan, folded at PhysicalPlan level

### Verdict: Approach C (Hybrid) вЂ” Unanimous across all 4 agents

```
LogicalPlan:   Limit в†’ Sort в†’ Project в†’ Filter в†’ Scan     (separate, composable)
PhysicalPlan:  IndexScan(table, range: age>18, limit: 10)  (folded, redb-aware)
```

---

### 8.1 DBSP Math: Why Sort and Limit CANNOT Live in Scan

**Filter (WHERE)** is a linear DBSP operator: `D[Пѓ_p(R)] = Пѓ_p(D[R])`. Zero state, fully
incremental. Each incoming `(record, weight)` is tested independently. This is the easiest
operator вЂ” it can live anywhere in the tree.

**Sort (ORDER BY)** is NOT linear: `D[sort(R)] в‰  sort(D[R])`. The correct incremental
formulation is `sort(R + О”R) в€’ sort(R)`, which requires materializing the full sorted
state. Sort must maintain a `BTreeMap` over ALL records (not just the current batch).
It cannot be a simple iterator decorator on redb in the streaming case.

**Limit** is the MOST non-incremental operator. Example:

```
State: {alice, bob, charlie}  в†’  LIMIT 3 output: {alice, bob, charlie}

Insert "aaron":
  Output becomes: {aaron, alice, bob}
  Delta: (aaron, +1), (charlie, -1)  в†ђ charlie was EVICTED

Delete "alice":
  Output becomes: {aaron, bob, charlie}
  Delta: (alice, -1), (charlie, +1)  в†ђ charlie RE-ENTERS
```

A single insertion can cause a retraction of an unrelated record. A deletion can
cause an insertion. The Sort+Limit operator must maintain ALL records ever seen
(not just top-N) because evicted records may return:

```rust
pub struct SortLimitOp {
    all_records: BTreeMap<SortKey, (Record, Weight)>,  // O(total records) memory
    current_output: BTreeSet<(SortKey, Record)>,       // current top-N
    limit: usize,
}
```

**Consequence**: Embedding Sort+Limit into Scan is semantically wrong for streaming.
The Scan receives CDC events one at a time вЂ” it cannot maintain the global sorted state
needed for correct Limit behavior. Sort+Limit must be separate stateful operators.

**Exception**: For one-shot batch queries (not streaming), redb's B-tree order
CAN be exploited: `table.range(19..).take(10)` is O(10), not O(N). But this is a
physical optimization, not a logical one.

---

### 8.2 The Three Layers

| Layer | Responsibility | WHERE | ORDER BY | LIMIT |
|-------|---------------|-------|----------|-------|
| **LogicalPlan** | Algebraic intent | `Filter` node | `Sort` node | `Limit` node |
| **PhysicalPlan** | Algorithm selection + redb awareness | `IndexScan.range` or `Filter` | Eliminated if matches key, else `Sort` | Folded into `Sort` or `IndexScan.scan_limit` |
| **Operator DAG** | Push-based execution | `FilterOp` (stateless) | `SortLimitOp` (BTreeMap state) | Part of `SortLimitOp` |

**LogicalPlan stays separate** вЂ” this is what the optimizer works on. Predicate pushdown,
projection pushdown, filter splitting all require separate nodes.

**PhysicalPlan folds** вЂ” `select_physical_plan` has access to `TableSchema` (which column
is the redb key). It makes storage-aware decisions:

```
Filter(age > 18, Scan("user"))  в†’  IndexScan("user", range: age > 18)     // key predicate
Filter(name = "x", Scan("user"))  в†’  Filter(name="x", IndexScan("user"))  // non-key, stays
Sort(age ASC, IndexScan)  в†’  eliminated (matches redb key order)
Sort(name ASC, IndexScan)  в†’  Sort(name ASC, IndexScan)  (does NOT match key)
Limit(10, IndexScan)  в†’  IndexScan(scan_limit: 10)  (only if Sort was eliminated)
Limit(10, Sort(...))  в†’  Sort(limit: Some(10))  (fused as today)
```

### 8.3 Concrete Types for the Parser AST

Extend `SelectStatement` in `src/parser/types.rs`:

```rust
#[derive(Debug, Clone)]
pub struct SelectStatement {
    pub fields:     Vec<Field>,
    pub table:      String,
    pub where_:     Option<WhereExpr>,     // NEW
    pub order_by:   Vec<OrderByKey>,       // NEW (empty = no ORDER BY)
    pub limit:      Option<u64>,           // NEW
    pub start:      Option<u64>,           // NEW (SurrealQL's OFFSET)
}

// в”Ђв”Ђв”Ђ WHERE clause в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ

#[derive(Debug, Clone)]
pub enum WhereExpr {
    BinaryOp { op: WhereOp, left: Box<WhereOperand>, right: Box<WhereOperand> },
    And(Box<WhereExpr>, Box<WhereExpr>),
    Or(Box<WhereExpr>, Box<WhereExpr>),
    Not(Box<WhereExpr>),
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum WhereOp { Eq, Ne, Gt, Ge, Lt, Le, In }

#[derive(Debug, Clone)]
pub enum WhereOperand {
    Column(String),
    LiteralInt(i64),
    LiteralFloat(f64),
    LiteralStr(String),
    LiteralBool(bool),
    LiteralNull,
    Subquery(Box<SelectStatement>),
}

// в”Ђв”Ђв”Ђ ORDER BY в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ

#[derive(Debug, Clone)]
pub struct OrderByKey {
    pub column: String,
    pub direction: SortDir,
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum SortDir { Asc, Desc }
```

**Why separate from `Predicate`?** `WhereExpr` can hold `Subquery(Box<SelectStatement>)` вЂ”
needed for `col IN (SELECT ...)` decorrelation. `Predicate` (in plan.rs) is post-decorrelation
and has no subquery variant. The subquery must survive into `lower_select` so decorrelation
can fire.

**Why `f64` not `OrderedFloat`?** The parser has no need for `Hash + Eq` on floats. The
lowering step wraps it: `LiteralFloat(f) в†’ Value::Float(OrderedFloat(f))`.

**`SortDir`**: Can be shared with `plan::SortDir` to avoid duplication. It's a pure semantic
value with no syntax-specific content.

### 8.4 Lowering: Node Stacking Order

`lower_select` produces this plan tree (innermost to outermost):

```
Limit { count: 10, offset: 5 }
  в””в”Ђв”Ђ Sort { keys: [("name", Asc)] }
        в””в”Ђв”Ђ Project { columns: ["name", "age"] }
              в””в”Ђв”Ђ Filter { predicate: Eq("active", Bool(true)) }
                    в””в”Ђв”Ђ Filter { predicate: Gt("age", Int(18)) }
                          в””в”Ђв”Ђ Scan { table: "user" }
```

For `SELECT name, age FROM user WHERE age > 18 AND active = true ORDER BY name ASC LIMIT 10 START 5`.

The order matches SQL semantics:
1. **Scan** вЂ” raw source
2. **Filter(s)** вЂ” WHERE predicates (AND produces stacked Filters)
3. **Project** вЂ” SELECT list column reduction
4. **Sort** вЂ” ORDER BY (needs projected columns to still include sort keys)
5. **Limit** вЂ” LIMIT/START always outermost

```rust
pub fn lower_select(stmt: SelectStmt) -> Result<LogicalPlan, LoweringError> {
    let mut plan = LogicalPlan::Scan { table: stmt.table };

    // WHERE в†’ Filter / Join (decorrelation for IN subqueries)
    if let Some(cond) = stmt.where_clause {
        plan = lower_cond(cond, plan)?;
    }

    // SELECT fields в†’ Project
    if !stmt.select_fields.is_empty() {
        plan = LogicalPlan::Project {
            input: Box::new(plan),
            columns: stmt.select_fields,
        };
    }

    // ORDER BY в†’ Sort
    if !stmt.order_by.is_empty() {
        plan = LogicalPlan::Sort {
            input: Box::new(plan),
            keys: stmt.order_by.into_iter()
                .map(|k| (k.column, k.dir))
                .collect(),
        };
    }

    // LIMIT / START в†’ Limit
    if stmt.limit.is_some() || stmt.start.is_some() {
        plan = LogicalPlan::Limit {
            input: Box::new(plan),
            count: stmt.limit.unwrap_or(usize::MAX as u64) as usize,
            offset: stmt.start.unwrap_or(0) as usize,
        };
    }

    Ok(plan)
}
```

### 8.5 PhysicalPlan: IndexScan with redb Awareness

```rust
pub struct ScanParams {
    pub table: String,
    /// Key range extracted from a Filter predicate on the redb key column.
    /// None = full table scan.
    pub range: Option<KeyRange>,
    /// Limit pushed into the redb iterator: `.take(n)`.
    /// Only valid when the scan order matches ORDER BY (Sort was eliminated).
    pub scan_limit: Option<usize>,
    /// What ordering the scan output provides.
    pub order: OrderGuarantee,
}

pub struct KeyRange {
    pub col: String,
    pub lo: Option<(Value, bool)>,  // (bound, inclusive)
    pub hi: Option<(Value, bool)>,
}

pub enum OrderGuarantee {
    None,
    Sorted(Vec<(String, SortDir)>),
}

pub enum PhysicalPlan {
    IndexScan(ScanParams),           // replaces Source
    Filter { input: Box<PhysicalPlan>, predicate: Predicate },  // residual only
    Project { input: Box<PhysicalPlan>, columns: Vec<String> },
    SymmetricHashJoin { /* ... */ },
    Sort { input: Box<PhysicalPlan>, keys: Vec<(String, SortDir)>, limit: Option<usize> },
    // ...
}
```

**The combined optimization trace** for
`SELECT name FROM user WHERE age > 18 ORDER BY age LIMIT 10` (age = redb key):

```
Step 1 вЂ” Scan("user")
  в†’ IndexScan(range: None, order: Sorted([("age", Asc)]))

Step 2 вЂ” Filter(age > 18)
  в†’ predicate_to_range(Gt("age", 18), key_col="age") в†’ range consumed, no residual
  в†’ IndexScan(range: age > 18, order: Sorted([("age", Asc)]))

Step 3 вЂ” Sort(age ASC)
  в†’ scan_order_satisfies([("age",Asc)], [("age",Asc)]) в†’ true
  в†’ Sort ELIMINATED. IndexScan passes through.

Step 4 вЂ” Limit(10)
  в†’ input is bare IndexScan в†’ set scan_limit = 10
  в†’ IndexScan(range: age > 18, scan_limit: 10, order: Sorted)

Runtime: table.range(Excluded(18)..).take(10)  вЂ” reads exactly 10 B-tree entries
```

vs naive: scan all 1M rows, filter 820K, sort remaining 180K, take 10. **~10,000x faster**.

### 8.6 When redb Optimization Does NOT Apply (Streaming Case)

For continuously maintained views via CDC events, the redb B-tree optimization only helps
with the **initial snapshot population**. Incremental maintenance always requires:

- `FilterOp`: stateless, zero overhead вЂ” works identically for batch and streaming
- `SortLimitOp`: stateful BTreeMap over ALL records вЂ” cannot be replaced by redb iteration
  because CDC events arrive one at a time, not as a sorted batch

The practical architecture:

```
Batch query (one-shot):   IndexScan(range, limit)  в†’  redb does the work
Streaming view (CDC):     FilterOp + SortLimitOp    в†’  in-memory state does the work
```

Both paths share the same LogicalPlan. The `select_physical_plan` function chooses
between them based on whether the query is a one-shot scan or a maintained view.

### 8.8 Decision Summary

| Question | Answer |
|----------|--------|
| Separate nodes or push into Scan? | **Separate in LogicalPlan, fold in PhysicalPlan** |
| Where does redb awareness live? | **`select_physical_plan`** (not optimizer, not runtime) |
| Can Filter be pushed into redb? | **Yes** вЂ” key predicates become `table.range()` |
| Can Sort be eliminated? | **Yes** вЂ” when ORDER BY matches redb key order |
| Can Limit be pushed into redb? | **Yes** вЂ” but ONLY when Sort was already eliminated |
| What about streaming/CDC? | Sort+Limit always need in-memory `SortLimitOp` state |
| Parser AST types вЂ” shared or separate? | **Separate** `WhereExpr`/`WhereOp`/`WhereOperand` (subquery support) |
